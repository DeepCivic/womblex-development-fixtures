## Enrichment Fixtures

Enrichment fixtures contain real Isaacus API responses (kanon-2-enricher) for each chunk in the corresponding chunk fixtures. Each enrichment result follows the `EnrichmentResult` dataclass structure from `src/womblex/analyse/models.py`.

### Structure

```json
{
  "fixture_id": "mso-statistics",
  "description": "Fuel MSO statistics (7 sheets)",
  "source_document": "_spreadsheets/mso-statistics-sept-qtr-2025.xlsx",
  "stats": {
    "documents_enriched": 7,
    "total_persons": 12,
    "total_locations": 5,
    "total_dates": 3,
    "total_external_docs": 2
  },
  "enrichments": [
    {
      "text": "document text...",
      "type": "other",
      "jurisdiction": "AU",
      "segments": [...],
      "persons": [...],
      "locations": [...],
      "terms": [...],
      "emails": [],
      "websites": [],
      "phone_numbers": [],
      "id_numbers": [],
      "external_documents": [],
      "quotes": [],
      "dates": [],
      "headings": [],
      "junk": [],
      "document_id": "..."
    }
  ]
}
```

Each enrichment object in the `enrichments` array corresponds to a non-empty chunk and contains the full ILGS (Isaacus Legal Graph Schema) response:
- **text**: The full document text (echoed back by the API)
- **type**: Document type (statute, regulation, decision, contract, other)
- **jurisdiction**: Jurisdiction code (AU for Australia)
- **segments**: Hierarchical document structure segments with spans
- **persons**: Person entities (natural, corporate, politic) with name spans and mentions
- **locations**: Location entities with name spans and mentions
- **terms**: Defined terms with name and meaning spans
- **emails**, **websites**, **phone_numbers**, **id_numbers**: Contact information
- **external_documents**: References to legislation and other documents
- **quotes**: Quotations within the text
- **dates**: Date information with type classification
- **headings**: Heading spans
- **junk**: Spurious text spans

All entity spans are zero-based, half-open Unicode code-point indices into the document text, as returned by the Isaacus API.

## Embedding Fixtures

Embedding fixtures contain real vector embeddings generated by the Isaacus kanon-2-embedder API for each chunk in the corresponding chunk fixtures.

### Structure

```json
{
  "fixture_id": "mso-statistics",
  "description": "Fuel MSO statistics (7 sheets)",
  "source_document": "_spreadsheets/mso-statistics-sept-qtr-2025.xlsx",
  "embedding_config": {
    "model": "kanon-2-embedder",
    "dim": 1792,
    "normalized": true
  },
  "stats": {
    "total_embeddings": 42
  },
  "embeddings": [
    {
      "chunk_id": "document_id:chunk:0",
      "document_id": "...",
      "chunk_index": 0,
      "embedding": [0.123, -0.456, ...]
    }
  ]
}
```

Each embedding object contains:
- **chunk_id**: Unique identifier matching the chunk from the chunk fixture
- **document_id**: Source document identifier
- **chunk_index**: Index within the document's chunks
- **embedding**: 1792-dimensional L2-normalised vector (kanon-2-embedder default)

## Generation

Fixtures are generated by `scripts/generate_enrichment_fixtures.py` which:
1. Reads chunk fixtures from `fixtures/womblex-collection/_chunks/`
2. Calls the real Isaacus API (kanon-2-enricher) for each chunk's text
3. Calls the real Isaacus API (kanon-2-embedder) for each chunk's text
4. Writes enrichment and embedding fixtures to this directory

### Prerequisites

```bash
# Isaacus SDK
pip install womblex[isaacus]

# Set your API key in .env
echo 'ISAACUS_API_KEY=your-key-here' > .env
```

### Regenerating

```bash
uv run python scripts/generate_enrichment_fixtures.py
```

## Usage in Tests

These fixtures are used to test:
- Enrichment result parsing and conversion
- Graph construction from enrichment results
- Entity span fidelity (all spans decode to real substrings)
- Embedding normalisation and dimensionality
- Consistency between enrichment/embedding counts and extracted/chunk fixtures

## Protection Guidelines

**⚠️ IMPORTANT: These fixtures are expensive to recreate and should be protected:**

### Why They're Expensive
- **API Costs**: Each enrichment call to Isaacus kanon-2-enricher incurs API costs
- **Embedding Costs**: Each embedding generation via kanon-2-embedder consumes API credits
- **Time Investment**: Regenerating all fixtures takes hours of API calls
- **Consistency**: Recreating fixtures may produce slightly different results due to model updates

### Protection Measures
1. **Version Control**: Always commit changes to these fixtures
2. **Backup**: Keep backups of the `fixtures/isaacus/` directory
3. **Regeneration Caution**: Only regenerate when absolutely necessary
4. **Partial Regeneration**: Consider regenerating only specific fixtures if needed
5. **API Key Management**: Keep `ISAACUS_API_KEY` secure and monitor usage

### When to Regenerate
- **Required**: When chunk fixtures change significantly
- **Optional**: When testing new Isaacus API features
- **Avoid**: For routine testing or development

## Notes

- Enrichment data is produced by the real Isaacus API — entity spans reference real positions in the document text
- Embeddings are produced by the real Isaacus kanon-2-embedder API
- Regeneration requires a valid `ISAACUS_API_KEY` and network access
- The committed fixtures can be used for testing without API access
- Separate MiniLM (all-MiniLM-L6-v2, 384-dim) embeddings are in `fixtures/MiniLM_embeddings/`, generated by `scripts/generate_minilm_embeddings.py`

## Graph Fixture

A single JSONL property graph combining auditor-general, dfat-budget-statements, and mso-statistics documents. Each line is a JSON object representing either a node or an edge.

### Node format

```json
{"type": "node", "fixture_id": "auditor-general", "chunk_index": 0, "id": "auditor-general:chunk:0:per:0", "label": "person", "properties": {"name": "...", "type": "corporate", "role": "..."}}
```

### Edge format

```json
{"type": "edge", "fixture_id": "auditor-general", "chunk_index": 0, "source": "...", "target": "...", "relation": "mentions", "properties": {"start": 100, "end": 150}}
```

### Fields

- `fixture_id`: Source fixture (auditor-general, dfat-budget-statements, mso-statistics)
- `chunk_index`: Index into the enrichment array for that fixture
- `label`: Node type (document, chunk, segment, person, location, term, external_document)
- `relation`: Edge type (contains, parent_of, mentions, mentioned_in, resides_at, cites, etc.)

### Generation

```bash
uv run python scripts/generate_graph_fixtures.py
```

### Core Integration

The graph generation script now uses the core Womblex API instead of duplicating parsing logic:

- Uses `EnrichmentResult.from_dict()` factory method from `analyse/models.py`
- Uses `DocumentGraph.to_dict()` serialization from `analyse/graph.py`
- All graph construction logic is now part of the public API in `analyse/__init__.py`
- Span utilities (`offset_span()`, `offset_spans()`) available for custom graph processing
